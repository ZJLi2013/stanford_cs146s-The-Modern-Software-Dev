## NotebookLM 风格总结 / 概念图

### 核心概念
```
Agent 部署之后
├── 分级与调试（Triaging and debugging）
├── 自动化事件响应（Automated incident response）
└── AI 系统的监控与可观测性（Monitoring and observability for AI systems）
```

### 要点列表
- **分级与调试**：当 Agent 或 AI 增强系统在生产出问题时，如何快速定位是模型、数据、提示还是下游依赖；建立分类与升级路径，避免「黑盒」式排查。
- **自动化事件响应**：用 AI/Agent 参与 on-call、告警分析、日志归纳、建议修复或自动回滚；Resolve 等产品聚焦「AI 原生 on-call」与多 Agent 协作。
- **可观测性**：为 AI 系统建立 traces、spans、指标与日志的最佳实践；理解「上下文衰减」、延迟与成本在运维中的影响。
- **多 Agent 与工程师角色**：多 Agent 系统如何分工（诊断、修复、通知）、如何与人类 on-call 工程师协作；软件工程师如何借 AI 成为「AI-native」运维。

### 关键词
`Post-deployment` `Incident response` `DevOps` `On-call` `Observability` `Resolve` `SRE`

### 可选 FAQ
- **Q：AI 能完全替代 on-call 吗？**  
  A：当前更适合做首轮分类、日志归纳与建议动作；升级策略、客户沟通与责任归属仍需要人；目标是「AI 先筛一遍，人做关键决策」。
- **Q：可观测性对 AI 系统有什么特别？**  
  A：除了传统指标外，需关注 prompt/上下文长度、模型延迟、token 消耗与错误类型分布，便于做成本与可靠性权衡。

## 讲师 / 对话总结

第 9 周讨论**部署后的运维与事件响应**：周一讲事件响应与 DevOps——如何把分级、调试与自动化嵌入到含 AI/Agent 的系统；周五由 **Resolve** 的 Mayank Agarwal（CTO）与 Milind Ganjoo（Technical Staff）分享 on-call 中的 Agentic AI、多 Agent 协作，以及 Kubernetes 故障排查与 AI 的结合。核心信息：Agent 与 AI 不仅改变开发阶段，也改变运维与 on-call；可观测性、自动化响应与人的升级路径要一起设计，才能既提效又可控。
