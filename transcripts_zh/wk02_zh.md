## 中文翻译（全文）

我们现在几乎在所有事情上都依赖大语言模型（LLM）：写代码、起草邮件、综合复杂研究。但这些强大的数字大脑面临一个根本挑战：一个仅基于过去静态数据训练的系统，如何与实时动态的世界进行有意义的交互？比如股票价格，或者像公司 Jira 看板那样位于安全 API 之后的数据。这是现代 AI 扩展的终极瓶颈。它们拥有渊博的世界知识，但这些知识在训练完成的那一刻就过时了。除非进行昂贵的重新训练，否则它们无法更新。如果你现在问 LLM 你最高优先级的 bug ticket 状态，或者主分支上最新的 git commit 是什么，它完全不知道。它被困在训练数据收集时的那个世界版本里。

要构建真正的自主 AI Agent——那些能管理工作流、查天气或运行诊断的实干家——我们需要稳健、标准化且安全的方式来输入动态实时信息。我们需要可靠的工具调用和 RAG（检索增强生成）能力，而且不能太贵或引入大量安全漏洞。历史上，将每个 LLM 应用连接到每个外部数据服务、API 或数据库（N 个服务），意味着要构建 M×N 个自定义集成。每个连接都需要独特的代码来处理 API 访问、OAuth 方案、错误处理和数据格式转换。这是碎片化和维护的噩梦。开发者不得不不断编写样板代码，仅仅是为了在模型和实际工具之间传输数据。

为了解决这个复杂性，**Model Context Protocol (MCP)** 应运而生。MCP 是一个开放协议，专为让系统以通用、统一的方式向 AI 模型提供上下文和暴露工具而设计。它将那个自定义、碎片化且昂贵的 M×N 系统替换为简单得多的 M+N 系统。可以把 MCP 想象成 AI Agent 的通用适配器，或者 AI 世界的 USB-C。你不再需要 50 个不同的专有连接器来让 LLM 与 GitHub、Stripe 和 Postgres 对话，只需要一个标准化的插头——MCP。这极大地减少了整个生态系统的摩擦。

我们今天的任务是深入解析这个标准。我们将研究其核心架构和通信的具体机制，探索迅速增长的专用 MCP 服务器生态系统，并从根本上理解该协议如何解决安全性和真正的互操作性这些不可协商的企业需求。

**核心机制与架构**

MCP 的核心是一个在 2024 年底出现的开放标准，建立在 JSON-RPC 2.0 之上。选择 JSON-RPC 2.0 是为了可靠性和简单性，确保通信是纯粹的请求-响应模式，高度可预测且易于解析，这对依赖结构的语言模型至关重要。它借鉴了软件开发领域的语言服务器协议（LSP）的概念，但不同于 LSP 的被动响应（等待用户输入代码），MCP 专为主动的 Agent 工作流设计，工具可以根据 Agent 的目标推动对话。

在这个握手过程中有几个关键角色：
1.  **Host（主机）**：用户交互的应用程序，如 Cursor 或 Claude Desktop。
2.  **MCP Client（客户端）**：嵌入在主机中的库，维护与服务器的连接会话。
3.  **MCP Server（服务器）**：关键的中介，构建在实际工具、数据源或 API 之上的轻量级软件。
4.  **Tool（工具）**：底层的函数，如 git pull 命令或复杂的 SQL 查询，是 AI 模型最终可以调用的对象。

**工作流程**

流程通常是这样的：客户端向 MCP 服务器调用 `tools/list` 方法，询问“你能为我做什么？”。服务器返回一个结构化的 JSON payload，描述它提供的每个工具（名称、摘要、输入参数的 JSON schema）。主机将这些工具定义注入到 LLM 的上下文（通常是系统提示词）中。当用户提问“我当前分支的状态是什么？”时，LLM 意识到它无法凭记忆回答，于是发出一个结构化的工具调用请求。MCP 客户端将其转换为精确的 MCP 请求发给服务器。MCP 服务器执行底层的 Git 函数，处理 API 密钥或凭证，运行操作，然后将结果以 JSON 响应返回给客户端。LLM Agent 从未接触也无需知道密钥，安全层被完全抽象在服务器-客户端关系中。这是一个干净、安全且可重复的循环。

这种模块化使得部署非常灵活。开发者可以用任何编程语言（Python, TypeScript, Go, Java 等）构建服务器，只要它能处理 JSON-RPC。

**传输层与生态系统**

MCP 支持多种传输层。对于本地进程，通常使用标准输入输出（Stdio）。对于云端托管 AI Agent 需要的远程通信，MCP 支持服务器发送事件（SSE），允许服务器持续推送更新并维护有状态会话，而无需客户端不断轮询。这对于像数据库迁移这样的长时间运行进程尤为重要。

这种灵活性和语言无关性引发了工具的爆发式增长。官方和社区提供了涵盖通用数据访问（如 Fetch）、文件系统、Git 操作、甚至内存服务的服务器。企业领域也有了针对 Jira, Confluence, GitHub, Cloudflare, Stripe, Postgres, MongoDB, Databricks 以及 AWS 和 Azure 服务的官方集成。社区还开发了针对区块链（Solana）、游戏（Chess.com）、浏览器自动化（Playwright）甚至特定领域（如巴西法律数据）的服务器。

**发现与注册**

为了解决工具发现的问题，MCP Registry 于 2025 年 9 月推出。它是所有公开可用 MCP 服务器的官方、集中式开放目录和 API，标准化了发现机制。它作为单一的事实来源，允许社区驱动的审核（可以屏蔽恶意服务器），确保信任基线。

**部署与安全**

部署方面，开发者可以将服务器部署为无认证（用于公共任务）或带认证和授权（用于企业安全）。早期的 MCP 实现缺乏标准化的 OAuth 机制，导致开发者不得不手动管理凭证（例如手动将 GitHub token 上传到服务器），这阻碍了其在云端多租户环境下的扩展。解决方案是集成 **OAuth 2.0** 作为授权的扩展标准。这带来了动态客户端注册、自动端点发现，最重要的是安全的授权和令牌管理。用户只需通过 OAuth 授权一次，系统就能保证 MCP 客户端只能访问用户明确允许的资源（使用短效令牌）。这终于为云托管 MCP Agent 带来了现代化的安全性和合规性。

**对比与总结**

与早期的 ChatGPT 插件相比，MCP 是开放标准，支持有状态会话和连续的上下文交互，而插件通常是专有的、单次调用的。与 LangChain 等框架相比，LangChain 是开发者面向的工具实现标准（在编译时定义逻辑），而 MCP 是模型面向的协议，支持运行时的动态工具发现。Agent 不需要预先包含特定工具的 SDK，只需知道 MCP 服务器的位置即可使用新工具。这解耦了 Agent 的逻辑与工具的实现，实现了真正的可扩展互操作性。

这意味着 MCP 正在将 AI 助手从受限于静态知识的孤立天才，转变为装备精良、能安全访问并操作真实世界数据的集成工程师。虽然管道问题已基本解决，但资料也提示了未来的潜在障碍：今天的模型在面对过多工具时推理能力会下降，且 API 返回的冗长结果可能迅速填满上下文窗口。这提出了一个重要问题：为了最大化标准化协议的潜力，开发者是否必须将 API 设计为“AI 原生”的——即提供简化、摘要或高度精选的数据集，而不是传统的僵化接口？
