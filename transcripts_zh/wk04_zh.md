## 中文翻译（全文）

编程正处于一个戏剧性的拐点。回顾软件开发的历史，我们从那些英勇的独行侠开发者，发展到由 NASA 等项目推动的大型专业团队。大约在 2023 年，我们进入了 AI 辅助编程时代，学会了与 AI 共同编码。而现在，大约在 2025 年，我们正进入下一个更激进的阶段：开发者的核心角色完全转变了。不再是单个人写代码，也不仅仅是与 Copilot 这样的工具逐行协作，而是**单个开发者管理多个高度自主的 AI Agent 的产出**。你正在从程序员（Coder）转变为**Agent 经理（Agent Manager）**。

本周的深度解读聚焦于正在推动这一变革的工具，特别是 **Claude Code**。它被描述为一个底层的、无偏见的命令行工具，用于 Agent 编程。它很灵活，但要把它变成高生产力平台，你需要一套系统。这就是我们今天的任务：让未来的 Agent 经理理解成功所需的新元技能——**上下文工程（Context Engineering）**。掌握这一技能的开发者，其生产力相比传统开发提升了 2.5 倍。在 Stack Auth 这样的 AI 原生公司，成功的衡量标准已经改变：**好的代码现在只是好的上下文的副作用**。

**上下文工程与新工作流**

为什么能有 2.5 倍的提升？因为范式变了。我们过去专注于 Prompt 工程（写一条完美的指令），但这对于试图执行多步计划的自主 Agent 来说完全不够。新的挑战是上下文工程：定义系统的整个知识库，设定行为约束，建立护栏，以便 Agent 能在不造成附带损害的情况下自主运行。你的代码仓库现在是人类和 Agent 的共享工作区，这意味着自然语言产物——设计文档、项目计划、架构指南——突然变得和代码本身一样重要。你是在为 Agent 储备“粮草”。

这创造了一个渐进的、层级化的工作流：
1.  **人类作为架构师**：定义高层业务需求和目标。
2.  **委托给设计 Agent**：根据需求起草详细的设计文档。
3.  **人类审查**：这是必不可少的关卡。修复有缺陷的设计比修复写好的代码便宜得多。
4.  **实施计划**：Agent 将设计转化为一系列任务。
5.  **实施与“Backstop”**：Agent 写代码，但必须通过**测试作为护栏**。必须有广泛的、已提交的失败测试作为客观约束。如果 Agent 无法通过现有的测试套件，它就不能提交代码。
6.  **审查与更新**：Agent 经理审计功能，并确保 Agent 更新所有相关文档（如 `README.md`、Schema），形成一个持续自我改进的系统。

**核心机制：CLAUDE.md**

在 Claude Code 中，管理上下文的核心机制是一个特殊文件：`CLAUDE.md`。这可以看作是 Agent 对该特定仓库的个性化说明书。Claude Code 会在任何会话开始时自动将此文件读入上下文。这里适合放置项目特定的规则：核心工具函数、构建流程、如何运行测试套件、常用的 bash 命令、仓库礼仪等。你可以用 `init` 命令快速生成一个初始文件。

维护这个文件是 Agent 经理的主要挑战。它必须像基础设施即代码（IaC）一样被视为一等公民。如果它与代码不同步，Agent 就会基于过时的假设行动，导致错误。团队通常会自动化维护或指定 Agent 定期审计文档。为了强制执行，可以使用“IMPORTANT”或“YOU MUST”等强调词。

**控制与安全**

Claude Code 直接在终端运行，能写文件、执行命令，这听起来有点吓人。因此它默认是保守的：任何修改系统的操作都需要显式批准。作为 Agent 经理，你可以通过权限管理来平衡摩擦与效率。例如，`allow edit` 允许 Agent 自行修改文件，或者更细粒度的 `bash git commit` 允许 Agent 在无需持续监督的情况下提交代码，但限制其访问敏感的生产工具。此外，Agent 继承你的 bash 环境，但如果你有自定义 CLI 工具，你需要显式告诉它或让它运行 help 命令来学习。

**成功的工作流模式**

1.  **探索-规划-编码-提交（Explore, Plan, Code, Commit）**：对于非琐碎问题，先让 Agent 阅读文件但不写代码，要求它制定计划。使用触发词如“Think”、“Think hard”或“Ultra think”来让模型分配更多计算预算进行内部推演，防止草率行事。
2.  **测试驱动的 Agent 开发（TDD）**：Claude 在有清晰客观约束时表现最好。指示 Claude 先写全面的测试，确认测试失败，提交测试，然后写代码直到测试通过（变绿）。这给了 Agent 经理完全的信心。
3.  **视觉迭代**：对于前端，拖入设计图，使用 Puppeteer MCP 让 Claude 截图其实时实现，并根据视觉目标迭代代码。

**多 Agent 与专业化**

单个 Agent 试图同时扮演编码者、审查者和测试者时会感到困惑。真正的收益来自**专业化和集成（Ensembles）**。
*   **验证环**：运行两个并行实例。一个 Claude 写代码，另一个（或通过 MCP 调用的 Gemini）专门负责审查或测试。Stack Auth 发现，第二模型（Gemini）在预先发现安全问题和微妙 Bug 方面比写代码的 Claude 更好。
*   **并行执行**：利用 `git worktrees` 让 Agent 在互不干扰的隔离工作区中同时处理独立任务。
*   **元编程框架**：如 SuperClaude 或 Awesome Claude Agents，自动化编排，配置专门的 PM Agent、深层研究 Agent 等，抽象掉管理复杂性。

**跨领域的应用**

这种变革不仅发生在软件团队。
*   **数据基建**：Kubernetes 集群故障时，团队将仪表盘截图和错误日志喂给 Claude Code，它结合视觉上下文和系统配置知识，瞬间诊断出 IP 地址耗尽问题并提供修复命令。
*   **增长营销**：非技术团队成员建立工作流，用子 Agent 生成数百个 Google 广告变体，实现 10 倍创意产出和更快的 A/B 测试。
*   **法律/非技术**：非开发者在 1 小时内为有语言障碍的家人构建了功能性的预测文本应用。
*   **安全工程**：通过指示 Claude 追踪堆栈跟踪的控制流，将事件解决时间从 15 分钟缩短到 5 分钟。

**总结**

开发者的核心技能已从编码语法转变为**管理复杂性**。你的工作是定义上下文、管理 Agent 并审计它们的工作。鉴于 Agent 的巨大力量和潜在的破坏范围，你最深厚的领域知识和业务规则现在被用于定义规则集和审计工作。你需要思考 6 个月后的事情：当 Agent 变得指数级更快、更自主时，你今天需要建立什么样的上下文、保障措施和 TDD 模式来确保质量？这是需要你最深层人类专业知识的工作。
