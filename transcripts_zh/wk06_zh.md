## 中文翻译（全文）

欢迎回到深度解读。如果你一直在关注 AI，你会知道新的前沿不再仅仅是 LLM 在“说话”，而是 LLM 在“做事”。我们谈论的是 **Agentic AI（代理化 AI）**：这些软件程序使用 LLM 作为大脑来自主运作。它们设定目标，并使用真实世界的工具、API、数据库和代码解释器来采取行动。这是一个关键的转变。随着这些系统走出实验室进入现实世界，处理客户服务、金融甚至医疗等敏感任务，它们带来了全新的、被放大的安全威胁。我们不再只是谈论旧式的越狱（Jailbreaks），而是系统性的高后果威胁，如远程代码执行（RCE）和大规模数据窃取。

今天的任务是深入探讨最新安全研究中的两个相互关联的领域。首先，我们将探索攻击者如何让这些自主 Agent“变节”；其次，我们将深入了解 LLM 本身在处理海量信息流时的一个根本性隐患——**上下文腐烂（Context Rot）**，它基本上破坏了整个系统的可靠性。

**第一部分：新的攻击面**

当你将 LLM 连接到工具时，你赋予了它行动的能力，它也立即继承了所有经典的软件风险（SQL 注入、SSRF、XSS）。但现在的影响被急剧放大了，因为是 AI 在以机器速度自主决定何时以及如何采取行动。

主要的攻击手段包括：
1.  **提示注入（Prompt Injection）**：依然非常有效，用于颠覆 Agent 的行为，使其泄露数据或违反护栏。
2.  **工具滥用（Tool Misuse）**：通过欺骗性 Prompt 操纵 Agent 滥用其外部能力。这使得以前难以触及的漏洞变得大门敞开。
3.  **远程代码执行（RCE）**：最可怕的一种。如果你诱骗能执行代码的 Agent 运行恶意指令，你就能获得对其整个环境（主机网络、文件系统）的未授权访问。

研究人员构建了一个模拟的投资顾问 Agent 环境来演示这些风险。该环境包括编排 Agent、新闻 Agent（可读取网页）和股票 Agent（有代码解释器）。
*   **SSRF 攻击**：攻击者无需编写自定义代码，只需滥用新闻 Agent 的网页读取工具。攻击者不让它读 CNN，而是指示它“去读取公司内部 IP 地址（如 192.168.0.25）的新闻”。Agent 只是试图帮忙，却变成了攻击者的内部间谍，扫描私有网络资源并将侦察数据传回。
*   **凭证窃取**：针对股票 Agent 的代码解释器。攻击者指示 Agent 运行 Python 脚本，递归搜索文件系统中具有高香农熵（Shannon entropy）的字符串（通常是 API 密钥）。为了绕过 LLM 的内部泄露防护，攻击者指示 Agent 将结果进行 **Base64 编码**。由于 Base64 看起来像随机字符，它轻松绕过了防护，实现了“隐藏在眼皮底下”的泄露。
*   **服务令牌窃取**：指示代码解释器查询云元数据端点（如 Google Compute Engine 的内部 Token URL），获取服务账户访问令牌，从而可能接管整个云基础设施。

此外，还有一个针对开发者工具（GitHub Copilot 和 VS Code）的重大漏洞 **CVE-2025-3773**，被称为“YOLO 模式漏洞”。攻击者只需在源文件中隐藏一个 Prompt 注入，指示 Copilot 在项目的 `settings.json` 中写入一行 `chat.tools.approve`。一旦加载，Copilot 进入“YOLO 模式”，禁用所有危险操作（如运行 Shell 命令）的用户确认，直接导致开发者机器上的 RCE。这表明 AI 可能无意中创造出一种自我传播的 AI 病毒。

**第二部分：上下文腐烂与不一致性**

为什么复杂的 AI 会中这些简单的计？答案在于 LLM 的短期记忆。我们假设有了巨大的上下文窗口，LLM 处理第 1 万条信息和第 1 条信息一样好，但研究表明并非如此。

可以将上下文窗口想象成一个又深又乱的抽屉。上面的东西好找，但压在底下的信息会被压缩、甚至完全丢失。这就是**上下文腐烂**。LLM 不断试图压缩信息以进行管理，这是一个有损过程。标准的“大海捞针”测试不足以衡量这一点，因为真实任务需要语义理解。研究发现：
*   **输入越长，性能越差**：即使任务难度不变。
*   **干扰与歧义**：加入相关但无关的信息（干扰项）会加剧性能下降。
*   **模型差异**：面对歧义，Claude 模型倾向于保守（拒绝回答），而 GPT 模型倾向于幻觉（自信地给出错误答案）。
*   **结构悖论**：令人惊讶的是，**打乱（Shuffling）**“草堆”（随机重新排序句子，破坏逻辑流）反而能提高模型性能。这表明 LLM 的注意力机制可能会被结构化的逻辑模式绊倒，而打乱能更均匀地分布关键事实。
*   **简单重复失败**：在长上下文中，模型甚至无法可靠地完成简单的单词序列重复任务。

这对安全工具（SAST）意味着什么？Semgrep 的研究测试了 Claude Code 和 OpenAI Codex 在大型 Python Web 应用中查找漏洞的能力：
*   **高噪音**：误报率高达 82-86%。每找到 5 个 Bug，有 4 个是错的。
*   **特定优势**：Claude Code 擅长发现 IDOR（不安全的直接对象引用），因为这需要上下文推理（识别哪里假设了授权但未强制执行）。
*   **致命弱点**：在需要严格数据流分析（Taint Analysis）的漏洞（如 SQL 注入、XSS）上表现极差，因为 LLM 难以跨多个文件追踪用户输入。
*   **非确定性**：对同一代码运行相同的 Prompt 会得到截然不同的结果（一次扫出 3 个，下次 6 个，再下次 11 个）。这种不一致性源于上下文腐烂和有损压缩，使得单一 AI 扫描提供的安全感可能是虚假的。

**防御策略**

鉴于威胁的严重性和 LLM 的脆弱性，必须采取分层防御：
1.  **Prompt 加固**：将 Prompt 视为源代码。使用 XML/JSON 标签严格隔离用户输入与系统指令。使用二级监控 Prompt 检查主 Agent 是否违规。
2.  **运行时内容过滤**：部署实时防御层，在指令执行前检查并拦截 Prompt 注入、工具滥用和数据泄露（如 Base64 编码）。
3.  **工具输入清洗**：工具绝不能信任 Agent 的输入。必须进行严格的类型验证、边界检查和特殊字符过滤。
4.  **沙盒化（Sandboxing）**：对于任何运行代码的 Agent 都是不可协商的。使用容器强制隔离，遵循最小权限原则，并严格限制容器网络（屏蔽云元数据 IP `169.254.169.254`，禁用危险系统调用）。

**总结**

Agentic AI 提供了在现实世界采取行动的变革性力量，但底层的 LLM 却出奇地脆弱和不一致。这种根本性的不一致使得构建真正稳健的安全性极具挑战。最后留给大家一个思考：如果一个不一致的 AI Agent 在你的代码中引入了新的安全漏洞，或者 AI 生成的补丁破坏了其他东西，谁该为此负责？
